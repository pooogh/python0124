# Продвинутые HTTP-запросы и REST API

## Что такое REST API?

**API (Application Programming Interface)** - это контракт, по которому две программы общаются друг с другом. Если бэкенд интернет-магазина хочет получить погоду, он обращается к API погодного сервиса.

**REST (Representational State Transfer)** - это самый популярный архитектурный стиль для создания API. Большинство современных сервисов (от ВКонтакте до Telegram) построены по принципам REST. 

Главные правила REST API, которые вам нужно знать:

1. **Все является "Ресурсом".** В REST мы работаем с сущностями (ресурсами): пользователи, посты, комментарии. У каждого ресурса есть свой уникальный URL.
   * *Плохой URL (не REST):* `site.com/get_all_users` или `site.com/create_post`
   * *Хороший URL (REST):* `site.com/users` или `site.com/posts`

2. **Действие определяется HTTP-методом (CRUD).** Чтобы управлять ресурсами, REST API использует стандартные методы HTTP. Это концепция **CRUD** (Create, Read, Update, Delete):
   * `POST` (Create) - создать новый ресурс.
   * `GET` (Read) - получить список ресурсов или один конкретный.
   * `PUT` / `PATCH` (Update) - обновить ресурс (PUT меняет целиком, PATCH - частично).
   * `DELETE` (Delete) - удалить ресурс.

3. **Отсутствие состояния (Stateless).** Сервер не запоминает вас между запросами. Каждый ваш запрос должен содержать всю необходимую информацию для его выполнения (например, специальный токен авторизации).

4. **Обмен данными в формате JSON.** Клиент и сервер общаются, перекидываясь текстовыми JSON-объектами, а не HTML-страницами.

Далее мы отработаем эти принципы на практике.

## Задача 1: Полный CRUD и отправка JSON (Payload)

**Где здесь REST:** В этой задаче вы выступите в роли клиента, который полноценно управляет ресурсом "Посты" (`/posts`), используя разные HTTP-методы. Вы на практике увидите, как один и тот же URL ведет себя по-разному в зависимости от того, какой метод вы использовали.

**Условия**
- Напишите корутину `update_resource()`.
- Сначала отправьте `POST` запрос на `https://jsonplaceholder.typicode.com/posts`, чтобы **создать** ресурс. Передайте словарь `{"title": "Новый пост", "body": "Текст", "userId": 1}`. 
  *Примечание: в `aiohttp` для отправки JSON используется аргумент `json=...`. Он сам переведет словарь в строку и добавит нужные заголовки.*
- Распечатайте ответ (сервер вернет созданный объект с `id: 101`).
- Затем отправьте `PATCH` запрос на `https://jsonplaceholder.typicode.com/posts/1` (здесь `1` - это ID ресурса, который мы хотим **обновить**), передав словарь `{"title": "Обновленный заголовок"}`. 
- Распечатайте ответ сервера.

**Конструкции**
- `session.post(url, json=...)`
- `session.patch(url, json=...)`

**Пример вывода:**
```text
Создан пост: {'title': 'Новый пост', 'body': 'Текст', 'userId': 1, 'id': 101}
Обновлен пост 1: {'userId': 1, 'id': 1, 'title': 'Обновленный заголовок', 'body': '...'}
```

## Задача 2: Токены (Bearer Token) и подмена заголовков

**Где здесь REST:** Это практическая реализация принципа **Stateless** (отсутствие состояния). Сервер вас не помнит. Чтобы получить доступ к закрытым данным, вы обязаны прикреплять к каждому своему запросу "пропуск" - токен авторизации (обычно это JWT-токен).

**Условия**
- Напишите корутину `fetch_secure_data()`.
- Сделайте `GET` запрос к защищенному ресурсу `https://httpbin.org/bearer`.
- Токены передаются в служебной части запроса - Заголовках (`Headers`). Создайте словарь заголовков и добавьте туда ключ `Authorization` со значением `Bearer secret_token_777`.
- Бонус: серверы часто блокируют запросы от Python-скриптов. Добавьте в заголовки ключ `User-Agent` со значением `Mozilla/5.0 (Windows NT 10.0; Win64; x64)`, чтобы притвориться реальным браузером.
- Распечатайте статус-код (должен быть 200) и JSON-ответ.

**Конструкции**
- Словарь заголовков: `headers = {"Authorization": "Bearer ...", "User-Agent": "..."}`
- Передача в запрос: `session.get(url, headers=headers)`

**Пример вывода:**
```text
Статус: 200
Ответ: {'authenticated': True, 'token': 'secret_token_777'}
```

## Задача 3: Работа с файлами (Бинарные данные)

**Теория:**
REST API может отдавать не только JSON, но и файлы (картинки, документы). Метод `await response.json()` вызовет ошибку при попытке прочитать картинку, так как это не текст. Файлы нужно читать как сырые байты, используя метод `await response.read()`, а затем сохранять на диск средствами Python.

**Условия**
- Напишите корутину `download_image()`.
- Сделайте `GET` запрос к `https://httpbin.org/image/jpeg`.
- Считайте бинарные данные ответа.
- Сохраните эти данные в файл `test_image.jpg` в папке со скриптом.

**Конструкции**
- Чтение байтов: `image_bytes = await response.read()`
- Сохранение в файл: 
  ```python
  with open("test_image.jpg", "wb") as f:
      f.write(image_bytes)
  ```

**Пример вывода:**
```text
Картинка успешно скачана и сохранена как test_image.jpg!
```

## Задача 4: Stateful-сессии (Автоматизация Cookies)

**Теория:**
Хотя REST требует, чтобы запросы были независимыми (Stateless), в реальном мире браузеры постоянно используют Cookies для поддержания сессий. Объект `aiohttp.ClientSession()` работает как ваш браузер: он автоматически запоминает все куки, которые присылает сервер, и подставляет их в следующие запросы.

**Условия**
- Напишите корутину `test_session()`.
- Создайте **одну** сессию.
- Сделайте `GET` запрос к `https://httpbin.org/cookies/set/my_cookie/qwerty12345`.
- Сделайте второй `GET` запрос к `https://httpbin.org/cookies` **используя ту же самую переменную `session`**.
- Распечатайте ответ второго запроса и убедитесь, что сервер увидел куку.

**Пример вывода:**
```text
Первый запрос (установка куки): 200
Второй запрос (проверка). Сервер видит куки:
{'cookies': {'my_cookie': 'qwerty12345'}}
```

## Задача 5: Пагинация (Сбор данных по страницам)

**Теория:**
Ни один реальный API не отдаст вам базу из миллиона товаров за один запрос - это "положит" и сервер, и ваш клиент. Данные отдаются частями (страницами). Вы должны сделать запрос на первую страницу, затем на вторую и так далее.

**Условия**
- Напишите корутину `fetch_pages()`.
- Вам нужно получить посты с ресурса `https://jsonplaceholder.typicode.com/posts`.
- С помощью цикла `for` сделайте 3 последовательных запроса (страницы с 1 по 3).
- В каждом запросе через словарь `params` указывайте номер страницы (`_page=i`) и лимит записей (`_limit=5`).
- Распечатайте количество полученных постов на каждой итерации.

**Конструкции**
- `params={"_page": i, "_limit": 5}`

**Пример вывода:**
```text
Страница 1 скачана: получено 5 постов.
Страница 2 скачана: получено 5 постов.
Страница 3 скачана: получено 5 постов.
Итого собрано: 15 постов.
```

# Анализ метаданных HTTP

### Задача 6: Чтение заголовков ответа (Response Headers)

**Теория:**
Вместе с телом ответа (JSON или картинкой) сервер всегда присылает метаданные в виде словаря заголовков (Response Headers). Из них можно узнать, какую технологию использует сервер или какой тип данных он вам прислал.

**Условия**
- Напишите корутину `check_headers()`.
- Сделайте `GET` запрос к `https://httpbin.org/get`.
- Не скачивайте тело ответа (`.json()`). Вместо этого обратитесь к свойству `response.headers`.
- Извлеките и распечатайте значения двух ключей: `Content-Type` и `Server`.

**Конструкции**
- `response.headers.get("Ключ")`

**Пример вывода:**
```text
Тип контента: application/json
Используемый сервер: gunicorn/19.9.0
```

### Задача 7: Маршрутизация по статус-коду (Status Code Metadata)

**Теория:**
Статус-код - это главный маркер успешности запроса. Хорошо написанный бот или парсер никогда не падает с ошибкой. Он читает статус-код и принимает решение: повторить запрос, пропустить ссылку или вывести сообщение администратору.

**Условия**
- Напишите корутину `handle_statuses()`.
- Создайте список из 3 тестовых URL: 
  `https://httpbin.org/status/200`
  `https://httpbin.org/status/404`
  `https://httpbin.org/status/500`
- Пройдитесь по ним в цикле. При каждом запросе проверяйте свойство `response.status`.
- Если код 200 - печатайте "Успех". Если 404 - "Страница не найдена". Если 500 - "Внутренняя ошибка сервера".

**Конструкции**
- Свойство `response.status`
- Цепочка `if / elif`

**Пример вывода:**
```text
Запрос к /status/200: Успех
Запрос к /status/404: Страница не найдена
Запрос к /status/500: Внутренняя ошибка сервера
```

### Задача 8: Скрытая пагинация (X-Total-Count)

**Теория:**
В задаче №5 мы скачивали по 5 постов на страницу. Но как вашему фронтенду (или парсеру) понять, сколько всего страниц существует, чтобы нарисовать кнопки перелистывания? Многие API передают общее количество записей в базе данных не в самом JSON, а в невидимом заголовке ответа, таком как `X-Total-Count`.

**Условия**
- Напишите корутину `get_total_count()`.
- Сделайте `GET` запрос к `https://jsonplaceholder.typicode.com/posts`, передав параметры `_page=1` и `_limit=5`.
- Прочитайте полученный JSON-ответ и выведите длину полученного списка (будет 5).
- Затем прочитайте из заголовков ответа (`response.headers`) значение ключа `x-total-count` и тоже выведите его.

**Конструкции**
- Совмещение `await response.json()` и `response.headers.get('x-total-count')`

**Пример вывода:**
```text
Постов получено в ответе: 5
Всего постов в базе сервера: 100
```
